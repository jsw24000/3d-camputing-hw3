# CODEX 工作原则

本项目对应作业“三维场景多表示重建与评估”。后续所有代码、实验、记录和报告准备工作都应优先遵守本文档，避免长流程中偏离作业要求。

## 1. 作业目标

1. 核心目标是完成同一真实静态三维场景的多表示重建、比较和分析，不是单纯追求最好看的渲染结果。
2. 至少比较三种不同三维表示。本项目默认使用：
   - COLMAP sparse point cloud / geometry
   - Nerfacto / NeRF
   - Official 3D Gaussian Splatting
   - VGGT-GS 作为额外比较方法
3. 比较维度应覆盖视觉质量、几何质量、效率、资源消耗、可编辑性、拓扑灵活性、数据采集难度和机器学习集成能力。
4. 结论必须回答“每种表示学到了什么、丢掉了什么、适合什么场景”，不能只罗列指标。

## 2. 数据原则

1. 数据必须来自本人或本小组自行拍摄/采集，不能直接使用公开 demo 数据集替代。
2. 场景应是单个真实、静态、可复现的场景。
3. 推荐采集 50 到 150 张图像，或 30 到 90 秒视频后抽帧。当前视频已抽取 51 帧。
4. 图像之间应有充分重叠，建议相邻视角重叠超过 60%。
5. 至少保留约 10% 图像作为 hold-out test views，不参与训练，只用于 PSNR/SSIM/LPIPS 等评估。
6. 必须记录拍摄设备、分辨率、采集路线、训练/测试划分和预处理操作。

## 3. 方法原则

1. 三种主方法应代表不同表示假设，而不是同一代码库的几个参数变体。
2. 允许使用开源代码、软件和工具，但报告中必须写清来源、版本、关键命令、配置和改动。
3. 若硬件或依赖有限，可以降低分辨率、减少迭代次数或使用轻量替代流程，但要记录原因。
4. 不要把 COLMAP、VGGT 或任何单一方法的输出当成绝对真值。若用作 proxy 评估，必须说明偏差和局限。

## 4. 实验流程原则

1. 所有方法共享同一份抽帧图像、同一 train/test split 和同一评估口径。
2. 主流程优先使用 COLMAP 估计相机位姿；VGGT 作为额外方法或替代位姿来源对比。
3. 每个方法都应有清晰的训练/优化配置、输出目录、日志和可复现实验命令。
4. 任何失败、退化、训练不稳定或几何异常都要保留记录；失败案例本身也是报告材料。

## 5. 评估与记录原则

1. 新视角合成质量至少记录可获得的 PSNR、SSIM、LPIPS；若 LPIPS 无法安装，需说明原因并至少提交 PSNR/SSIM。
2. 效率与资源至少记录训练时间、迭代步数、GPU/CPU/VRAM 使用、模型或输出文件大小、渲染速度。
3. 几何质量至少做一种分析，例如点云/相机轨迹可视化、深度/法线/Gaussian 分布观察，或针对桌面、墙面等结构的人眼分析。
4. 指标不能替代解释。报告必须说明为什么某方法更清晰、更模糊、更稳定或更容易产生伪影。
5. 保留训练视角、hold-out test views、不同方法 qualitative comparison 和几何输出截图。

## 6. 报告内容原则

报告建议包含：

1. 摘要：说明场景、三种表示和核心发现。
2. 数据采集：场景、路线、图像数量、训练/测试划分、位姿估计结果。
3. 方法：表示假设、初始化、优化目标、输入输出、关键配置和选择理由。
4. 实验设置：硬件、软件版本、分辨率、迭代数、评估脚本和计时方式。
5. 结果：定量表格与定性对比图。
6. 讨论：结合课程维度分析不同表示的差异。
7. 结论：推荐的表示方法、适用条件、采集改进方向和下一步扩展。
8. 附录：关键命令、配置、代码链接或 commit、额外图表和失败案例。

## 7. 提交原则

最终提交应包含压缩包和一份报告 PDF。建议目录结构：

```text
student_id_name_assignment/
  report.pdf
  README.md
  config/
  scripts/
  results/
    metrics.csv
    qualitative/
    geometry/
  logs/
```

`README.md` 至少说明运行环境、复现实验命令、数据路径、每种方法的输出位置、训练/测试文件列表，以及大文件未提交时的替代说明。

## 8. 禁止偏差

1. 不要使用未经说明的外部数据冒充自己采集的数据。
2. 不要只交截图而没有实验记录。
3. 不要没有 hold-out test views 就声称完成新视角合成评估。
4. 不要只比较同一方法的不同参数并称为三种表示。
5. 不要伪造指标、训练时间、硬件使用或失败原因。
6. 不要为了结果好看而改变不同方法的测试集或评估口径。
7. 不要删除失败结果，应将其作为分析材料。

## 9. Codex 协作规则

1. 较长任务开始前，先对照本文档确认当前目标属于数据、方法、实验、评价、报告或提交物中的哪一类。
2. 涉及代码和实验脚本时，优先保持可复现：写清输入、输出、命令、配置和日志路径。
3. 涉及指标和结论时，必须区分“实际测量结果”“从图像观察得到的判断”和“基于文献/课程内容的解释”。
4. 如果 PDF、课程平台通知或用户新要求与本文档冲突，以用户最新要求和课程平台通知为准，并及时更新本文档。
5. 不确定作业要求时，先回到原始 PDF 或向用户确认，避免凭记忆继续推进。

## 10. 代码风格

1. 关键部分可以添加必要中文注释，尤其是模型架构、重要数据处理函数和 3DGS/VGGT 适配逻辑。
2. 脚本应支持 `--dry-run`、明确路径检查、日志输出和可复现的默认配置。
3. 默认使用 Conda 环境 `torch_env`；只有外部训练工具不兼容时，才创建方法专用备用环境。
